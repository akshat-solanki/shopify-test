    # Shopify App Architecture

## Purpose

This document defines the production architecture for the product showcase app without introducing routes, persistence implementation, or business logic.

The goal is to make the future backend safe for Shopify App Store launch:

- Shopify stays the source of truth.
- storefront rendering never depends on raw Shopify response shapes
- visual configuration stays JSON-based
- each dropped showcase instance remains isolated
- frontend and backend share a stable contract surface

## Local Dev Tunnel Strategy

For stable local Shopify development, this app should prefer a fixed custom tunnel over Shopify CLI's autogenerated temporary tunnel URLs.

Why:

- `shopify app dev` can generate a different public URL after every restart
- the embedded admin app depends on one consistent public app URL
- the theme extension can appear healthy even while the admin panel is broken, which makes temporary-tunnel drift misleading

### Current URL Dependencies

The public app URL is used in these places:

- `.env.local` as `APP_URL`
- `shopify.app.toml` as `application_url`
- `shopify.app.toml` auth redirect URL
- backend OAuth redirect generation in [server/shopify/oauth/oauth.service.ts](/Users/akshatsolanki/Downloads/Vypari Product Showcase Design/server/shopify/oauth/oauth.service.ts:88)
- backend auth redirect handling in [server/controllers/auth.controller.ts](/Users/akshatsolanki/Downloads/Vypari Product Showcase Design/server/controllers/auth.controller.ts:18)

### Stable Tunnel Workflow

1. Start a persistent custom tunnel yourself.
2. Put that fixed base URL into `.env.local` as `APP_URL`.
3. Put the same fixed base URL into `shopify.app.toml` as `application_url`.
4. Put the same fixed base URL into `shopify.app.toml` auth callback URLs.
5. Start Shopify with a custom tunnel:

```bash
shopify app dev --no-update --tunnel-url=https://your-stable-tunnel.example.com
```

Important:

- use the base host only, not `/proxy`
- keep `[app_proxy].url = "/proxy"` unchanged in `shopify.app.toml`
- do not let Shopify autogenerate and rotate the URL if you want a stable setup
- do not run a second unrelated temporary Cloudflare tunnel in parallel

### What Does Not Need To Change

No app business logic changes are required for the stable-tunnel approach.

The current backend already reads the public app URL from environment/config correctly.

No changes are needed in:

- OAuth service design
- app proxy routing
- storefront runtime loading
- theme extension placement logic

### Recommended Local Start Order

```bash
npm run start:server
shopify app dev --no-update --tunnel-url=https://your-stable-tunnel.example.com
```

## Source Of Truth Rule

The app must not permanently duplicate Shopify catalog entities.

Do not persist full copies of:

- products
- variants
- collections
- customers
- orders
- inventory
- reviews

Allowed:

- identifiers and references to Shopify resources
- short-lived caches with expiry
- normalized read models generated at request time
- analytics generated by this app
- configuration, presets, instances, experiments, validations, and installation metadata

## Core Domain Model

### Merchant

Top-level account owner inside our app. A merchant can manage multiple stores.

### Store

Represents one Shopify shop connected to the app. A store belongs to one merchant and holds Shopify-specific context such as domain, currency, locale, and time zone.

### Installation

Captures app install state, scopes, API version, and lifecycle state for a store.

### Subscription

Captures billing plan state for a merchant-store pair. This is app billing metadata, not Shopify commerce data duplication.

### Showcase

Merchant-owned logical showcase definition. Think of this as a reusable showcase family or workspace.

### Showcase Instance

A dropped storefront instance of a showcase. This is the most important runtime object.

Each instance is independent and stores:

- `instanceId`
- `merchantId`
- `configuration`
- `preset`
- `version`
- `status`
- `createdAt`
- `updatedAt`

Instances must never mutate one another.

### Configuration

Versioned JSON snapshot attached to a showcase, instance, or preset. This remains strongly typed in code, but persisted as one structured JSON object rather than many relational rows.

### Preset

Reusable merchant-facing starting point such as Fashion, Luxury, Beauty, Electronics, Furniture, Grocery, or Sports.

### Validation

Result of design and runtime checks against a showcase instance. This includes structured check output and summary counts.

### Analytics

Aggregated performance signals generated by the app for an instance:

- impressions
- clicks
- CTR
- add to cart
- add to cart rate
- quick view opens
- variant selections

### Experiment

Controlled comparison between showcase instances. Stores control and variant references plus primary success metric.

### Webhook Event

Durable record of Shopify webhook receipt and processing state. This supports reliability, replay, and auditability.

### Theme Configuration

Represents how an instance is attached to a Shopify theme surface or app embed placement.

## Entity Relationships

1. One `Merchant` has many `Store` records.
2. One `Store` has one active `Installation` and zero or one active `Subscription`.
3. One `Merchant` owns many `Showcase` records.
4. One `Showcase` has many `ShowcaseInstance` records.
5. One `ShowcaseInstance` resolves to exactly one active `ShowcaseConfiguration`.
6. One `Preset` can seed many `Showcase` or `ShowcaseInstance` records.
7. One `ShowcaseInstance` can have many `Validation` records over time.
8. One `ShowcaseInstance` can have many `Analytics` snapshots over time.
9. One `Experiment` points to one control instance and one or more variant instances.
10. One `ThemeConfiguration` connects one instance to one storefront placement.

## Normalized Shopify Product Contract

All Shopify product data must be normalized into one internal interface before reaching the UI.

The UI must never care whether a field came from:

- Storefront API
- Admin GraphQL API
- metafields
- variant-level data
- cached projection

The normalized shape is `ProductDomainModel` in [shared/contracts/index.ts](/Users/akshatsolanki/Downloads/Vypari Product Showcase Design/shared/contracts/index.ts:1).

### Product Coverage

The normalized contract supports:

- identity
- pricing
- media
- inventory
- commerce
- social proof
- taxonomy
- merchant extensions
- SEO
- descriptions
- delivery

### Normalization Rules

1. `price` and `compareAtPrice` are always returned as normalized `MoneyAmount`.
2. `discountPercentage` is computed into the internal contract so the UI never recalculates it ad hoc.
3. `featuredImage`, `galleryImages`, `videos`, `media`, and `models3d` are all exposed from one media set.
4. `variants`, `options`, and `optionValues` are normalized so variant pickers are backend-agnostic.
5. `pickupAvailability` is normalized into a simple location list.
6. `metafields` are exposed through a generic typed extension surface.
7. `rawReferences` may keep Shopify IDs for re-fetching, but not duplicate full Shopify entities permanently.

## Display Engine Shared Contracts

The display engine must be shared between backend and frontend through stable contracts.

Required shared contracts:

- `DisplayBlock`
- `AttributeRegistry`
- `CardVariant`
- `Density`
- `ThemeContract`
- `MotionContract`
- `BadgeContract`
- `CTAContract`
- `ValidationContract`
- `CommerceScoreContract`
- `MerchantConfidenceContract`

Why:

- backend can resolve configuration and validations consistently
- frontend can render without backend implementation details
- merchant presets stay portable across app surfaces

## Configuration Model

Visual configuration must remain one typed JSON object: `ShowcaseConfiguration`.

Top-level sections:

- `appearance`
- `commerce`
- `accessibility`
- `motion`
- `media`
- `cta`
- `theme`
- `validation`
- `quickView`
- `density`

This avoids an over-normalized schema while preserving type safety in code.

### Configuration Design Principles

1. Merchant intent first. Fields represent decisions such as CTA style, density, or image behavior, not raw CSS properties.
2. Surface-aware defaults. Grid, carousel, spotlight, and recommendation surfaces can override a small set of display rules.
3. Deterministic reduction. Compact cards remove lower-priority blocks first.
4. Safe override model. Presets can seed defaults, but instances own their final configuration.

## Service Boundaries

No service should own Shopify catalog truth. Services orchestrate configuration, normalization, validation, and app-owned metadata.

### Merchant Service

Responsibilities:

- fetch merchant summary
- list merchant stores
- resolve billing and account context

### Showcase Service

Responsibilities:

- create and update showcase records
- create independent showcase instances
- fetch showcase detail and instance detail
- enforce instance isolation and versioning

### Configuration Service

Responsibilities:

- resolve effective configuration
- merge preset defaults with instance overrides
- version configuration updates

### Preset Service

Responsibilities:

- list system and merchant presets
- create merchant presets
- apply presets to instances

### Validation Service

Responsibilities:

- run design QA checks
- emit structured validation results
- optionally compute commerce score and merchant confidence contracts

### Analytics Service

Responsibilities:

- aggregate app-owned performance metrics
- expose instance-level metrics for dashboards and experiments

### Experiment Service

Responsibilities:

- create and manage controlled showcase experiments
- define control and variant sets
- expose experiment outcomes using app-owned analytics

### Shopify Integration Service

Responsibilities:

- fetch Shopify resources on demand
- normalize Shopify responses into internal contracts
- manage cache lifetimes where required
- sync theme/app embed configuration

This service must be the only boundary that understands raw Shopify API shapes.

### Webhook Service

Responsibilities:

- ingest webhook envelopes
- store webhook event state
- trigger downstream refresh or invalidation workflows
- guarantee idempotent processing

## Caching Policy

Caching is allowed only to improve performance or rate-limit resilience.

Rules:

1. Cache normalized product projections, not raw long-term Shopify copies.
2. Every cache object must carry fetch time and source metadata.
3. Cache invalidation should be triggered by webhook events or TTL expiry.
4. Reads must tolerate cache misses gracefully.
